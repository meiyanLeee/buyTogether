# 社区团购优惠平台优化说明

## 1. 登录认证改为 Sa-Token

原项目使用自定义 Redis token 拦截器维护登录态，学习价值足够，但作为包装后的项目会显得偏跟练。现在改为 Sa-Token 统一签发和校验 token，前端统一通过 `satoken` 请求头携带 token。服务层仍通过 `UserHolder` 获取当前用户，避免影响点赞、关注、签到、抢券等业务代码。

这样做的好处是：

- 登录态管理交给成熟框架，减少手写续期和 token 校验代码。
- 面试时可以讲清“从自实现认证升级到权限框架”的演进思路。
- Controller 路径和前端存 token 的方式基本兼容，改造风险低。

## 2. 配置外置化

原项目把 MySQL、Redis、Redisson 的地址、账号和密码直接写在配置文件或 Java 配置类中。包装成可展示项目后，这类配置会暴露个人环境，也会让别人拉取项目后难以启动。

现在改为环境变量加本地默认值，例如：

- `MYSQL_URL`、`MYSQL_USERNAME`、`MYSQL_PASSWORD`
- `REDIS_HOST`、`REDIS_PORT`、`REDIS_DATABASE`、`REDIS_PASSWORD`
- `REDIS_POOL_MAX_ACTIVE`、`REDIS_POOL_MAX_IDLE`、`REDIS_POOL_MIN_IDLE`

这样既保留 Spring Boot + Redis 的原技术栈，也方便在本机、服务器和演示环境中切换。

## 3. Redis Stream 消费组初始化

原项目依赖手动执行测试方法创建 `stream.orders` 的 `g1` 消费组。如果忘记初始化，异步订单线程启动后可能持续报错，秒杀请求虽然返回成功，但数据库订单无法稳定落库。

现在在 `VoucherOrderServiceImpl` 初始化时尝试创建消费组。消费组已存在时只记录调试日志，不中断应用启动。这样演示时不再依赖额外手工步骤，异步下单链路更完整。

## 4. 团购券下单幂等

原 `createVoucherOrder` 在发现用户已有订单或库存扣减失败后只写日志，后续仍可能执行 `save(order)`。这会让“一人一单”和“库存不足不落库”的业务语义不够严谨。

现在新增 `VoucherOrderPolicy`，明确只有满足两个条件才创建订单：

- 当前用户还没有购买过该团购券。
- 数据库库存扣减成功。

该策略有纯单元测试覆盖，不依赖 MySQL 或 Redis，适合作为面试中解释幂等控制的代码证据。

## 5. 数据预热测试独立

缓存预热、GEO 数据导入、秒杀库存写入和 Stream 消费组创建都需要真实 MySQL/Redis。现在新增 `DataPreheatTests`，并默认标记 `@Disabled`，作为手动运行的本地预热工具。

这样普通 `mvn test` 不会因为外部服务不可用失败，需要演示时又可以单独运行这些预热方法。

## 6. 前端视觉重构

前端保持原生 Vue + Element UI + Nginx 静态页结构，不引入新的构建体系，降低改造风险。页面结构从“本地生活信息流”改成“社区团购小程序式体验”，重点重做：

- 首页：品牌头部、大图团购入口、场景品类、团长动态卡片。
- 列表页：自提点卡片、距离、跟团热度、评分筛选。
- 详情页：大图头图、自提说明、团购券、团友反馈。
- 登录页：图片背景和底部登录面板，强化社区团购第一印象。

## 7. 图片和种子数据替换

商户图片、团长动态图片、品类入口图标和主要头像都改为本地素材 `/imgs/community/photo*.png`，避免页面继续加载原项目外链图。SQL 中的帖子标题和正文也改成团购、自提、团长推荐语义。

这样导入数据后，首页、列表、详情和动态内容都能明显区别于常见跟练项目。

## 8. 后续可选优化

第一阶段没有引入运营后台、团长角色表或真实支付状态，避免改动过大。后续可以按优先级补充：

- 团长角色与团长认证字段。
- 团购订单状态：待支付、待自提、已核销、已退款。
- 自提点库存看板和团购券发布后台。
- 接口级集成测试与 Testcontainers 环境。
